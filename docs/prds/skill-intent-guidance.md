# PRD: 스킬 활용 의도 안내

---

## 문제

현재 시스템 프롬프트에 스킬 목록(`<skills>` XML)이 동적으로 주입되지만, SOUL.md의 의도 파악 테이블에 **스킬 활용** 유형이 없다. 두 가지 실패 패턴이 관찰된다:

### 패턴 1: 스킬을 아예 확인하지 않음

```
사용자: "이 영상 요약해줘"
  ↓
LLM: 의도 파악 테이블 확인 → "분석/요약" 또는 "파일/시스템 작업"으로 분류
  ↓
LLM: 서브에이전트(analyst) 위임 또는 직접 처리 시도
  ↓
결과: 스킬 목록에 summarize가 있지만 확인하지 않음
```

### 패턴 2: 스킬 사용을 말하고 실행하지 않음

```
사용자: "summarize 스킬로 이 영상 요약해줘"
  ↓
LLM: "summarize 스킬로 자막을 먼저 추출하겠습니다. 잠시만요." ← 텍스트만 생성
  ↓
LLM: web_search("영상 제목") 또는 직접 답변 시도 ← 스킬 실행 흐름 아님
  ↓
결과: 스킬 사용을 언급했지만 read_file(SKILL.md) → exec(CLI) 워크플로를 따르지 않음
```

LLM이 시스템 프롬프트에서 스킬 목록을 보고 스킬의 **존재**는 인지하지만, **실행 워크플로**(read_file → exec)를 따르지 않는다. 스킬명을 언급하면서도 실제로는 다른 도구로 대체 처리하거나 텍스트만 생성한다.

**문제점**:
1. 의도 파악 테이블에 "스킬 활용" 유형이 없어 LLM이 스킬 목록을 확인하는 흐름으로 연결되지 않음
2. 시스템 프롬프트의 스킬 안내("read_file로 SKILL.md를 읽으세요")는 **언제** 사용해야 하는지 말하지 않음
3. LLM이 스킬 존재를 인지해도 실행 워크플로(read_file → exec)를 명시적으로 안내받지 못해 텍스트만 생성하고 실행하지 않음
4. 스킬의 trigger phrases(예: "요약해줘")가 SKILL.md 내부에만 존재 — LLM이 먼저 읽어야만 알 수 있는 닭과 달걀 문제
5. 스킬은 동적으로 추가/삭제되므로 SOUL.md에 특정 스킬명을 하드코딩할 수 없음

## 해결책

SOUL.md의 의도 파악 테이블에 **스킬 활용** 유형을 추가한다. 특정 스킬명을 명시하지 않고, 시스템 프롬프트의 동적 스킬 목록을 참조하도록 제네릭하게 안내한다.

## 사용자 영향

| Before | After |
|---|---|
| "이 영상 요약해줘" → 스킬 무시, 직접 처리 시도 | 스킬 목록 확인 → summarize 스킬 발견 → SKILL.md 읽기 → 실행 |
| "날씨 알려줘" → 웹 검색 시도 | 스킬 목록 확인 → weather 스킬 발견 → curl로 즉시 조회 |
| "summarize 스킬로 요약해줘" → 말만 하고 실행 안 함 | read_file(SKILL.md) → exec(CLI) 워크플로 실행 |
| 스킬이 있는데 사용하지 않음 | 요청과 매칭되는 스킬을 우선 활용 |
| 스킬 추가/삭제 시 안내 변경 불필요 | 동일 (시스템 프롬프트가 동적으로 갱신) |

## 기술적 범위

- **변경 파일**: `shacs_bot/templates/SOUL.md` (1개)
- **변경 유형**: 마크다운 콘텐츠 추가 (코드 변경 없음)
- **의존성**: 없음. 기존 스킬 시스템(`skills.py`, `context.py`)에 의존하지만 코드 변경 없음.
- **하위 호환성**: 기존 의도 파악 동작에 유형 하나 추가. 기존 유형의 동작에 영향 없음.

### 변경할 내용 요약

**SOUL.md 의도 파악 테이블에 추가**:

```markdown
| 스킬 활용 | 요청이 시스템 프롬프트의 스킬 목록에 있는 스킬과 관련될 때 | 스킬 목록에서 매칭되는 스킬 확인 → read_file로 SKILL.md 읽기 → 지침대로 실행. |
```

위치: "기억 회상"과 "파일/시스템 작업" 사이. 스킬이 도구보다 상위 개념이므로 파일/시스템 작업보다 먼저 매칭되어야 한다.

### 설계 원칙

1. **제네릭 안내만 사용** — 특정 스킬명("summarize", "weather")을 SOUL.md에 하드코딩하지 않는다
2. **동적 스킬 목록 참조** — "시스템 프롬프트의 스킬 목록"이라는 표현으로 `context.py`가 주입하는 `<skills>` XML을 참조
3. **기존 유형과 충돌 방지** — "분석/요약" 유형과 겹칠 수 있으나, 스킬이 있으면 스킬 우선, 없으면 기존 대응

### 의도 파악 흐름 변화

```
Before:
  "이 영상 요약해줘" → 분석/요약 → analyst 위임 → 스킬 무시

After:
  "이 영상 요약해줘" → 스킬 목록 확인 → summarize 스킬 발견
    → read_file("...skills/summarize/SKILL.md")
    → exec("summarize 'https://...'")
    → 결과 반환

  "이 코드 분석해줘" → 스킬 목록 확인 → 매칭 스킬 없음
    → 분석/요약 → analyst 위임 (기존 동작 유지)
```

## 성공 기준

1. "영상 요약해줘" 같은 요청에서 봇이 스킬 목록을 확인하고 관련 스킬을 사용한다
2. 스킬 사용을 말로만 하지 않고 실제로 read_file(SKILL.md) → exec(CLI) 워크플로를 실행한다
3. 스킬이 없는 요청("이 코드 분석해줘")은 기존처럼 서브에이전트로 처리된다
4. 새 스킬을 추가하면 다음 메시지부터 자동으로 활용된다 (SOUL.md 수정 불필요)
5. 기존 의도 파악 유형(일상 대화, 단순 질문 등)의 동작에 영향 없다
6. 특정 스킬명이 SOUL.md에 하드코딩되어 있지 않다
